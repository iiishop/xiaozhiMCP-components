# 组件开发指南

本指南说明如何在 `xiaozhiMCP-components` 中新增或维护组件。目标是让组件具备清晰边界、统一接口、可被目录系统发现，并能在 server/client 或集群场景下稳定运行。

## 开发前判断

新增能力前先判断它是否应该是组件：

| 判断问题 | 推荐归属 |
| --- | --- |
| 是否依赖外部 API、外部 MCP 服务或独立凭据？ | 组件 |
| 是否强依赖 Windows、macOS 或 Linux 特定 API？ | 组件 |
| 是否拥有独立数据文件、缓存或迁移逻辑？ | 组件 |
| 是否是主系统必须存在的基础能力？ | 主仓库核心 |
| 是否只是核心模块的 30 到 50 行薄封装？ | 主仓库核心 |

## 标准目录

新增组件名建议使用小写字母、数字和下划线，目录名、`index.json` 的 `name` 和配置段名称保持一致。

```text
my_component/
  __init__.py
  component.py
  README.md
```

`__init__.py` 可以为空，也可以重新导出测试需要直接导入的类：

```python
from .component import MyComponent

__all__ = ["MyComponent"]
```

## 最小组件模板

```python
from __future__ import annotations

from typing import Any


class MyComponent:
    def __init__(self, config: dict[str, Any] | None = None, full_config: dict[str, Any] | None = None) -> None:
        self.config = config or {}
        self.full_config = full_config or {}

    def register(self, mcp: Any) -> None:
        @mcp.tool()
        def my_component_echo(text: str) -> dict:
            """Return the input text."""
            return {"success": True, "text": text}

    def export_tools(self) -> list[dict[str, Any]]:
        return [
            {
                "name": "my_component_echo",
                "description": "Return the input text.",
                "input_schema": {
                    "type": "object",
                    "properties": {
                        "text": {"type": "string", "description": "Text to return."}
                    },
                    "required": ["text"],
                },
            }
        ]

    def invoke_tool(self, tool_name: str, arguments: dict[str, Any]) -> dict:
        if tool_name == "my_component_echo":
            return {"success": True, "text": str(arguments.get("text", ""))}
        raise RuntimeError(f"unknown tool: {tool_name}")


def build_component(config: dict[str, Any] | None = None, full_config: dict[str, Any] | None = None) -> MyComponent:
    return MyComponent(config=config, full_config=full_config)
```

## 工具命名规范

工具名必须全局唯一，推荐使用组件名前缀：

| 组件 | 推荐工具名前缀 | 示例 |
| --- | --- | --- |
| `exa` | `exa_` | `exa_web_search` |
| `clipboard` | `clipboard_` | `clipboard_set` |
| `windows_manager` | `windows_` | `windows_focus_app` |
| `local_schedule` | `schedule_` | `schedule_add_event` |
| `apple_music_macos` | `apple_music_` | `apple_music_search` |

避免使用 `search`、`get`、`list` 这类无前缀名称，否则容易与核心工具或其他组件冲突。

## `export_tools()` 规约

`export_tools()` 是集群和远程注册的核心契约。每个条目必须包含：

```python
{
    "name": "my_component_action",
    "description": "清楚说明何时使用该工具。",
    "input_schema": {
        "type": "object",
        "properties": {
            "param": {
                "type": "string",
                "description": "参数说明"
            }
        },
        "required": ["param"]
    }
}
```

建议在 schema 中补充 `default`、`minimum`、`maximum`、`enum` 等约束。`exa_web_search` 就通过 `enum` 暴露搜索类型和内容模式，便于调用端生成稳定参数。

## `register()` 规约

`register(mcp)` 负责把工具挂到 FastMCP 上：

```python
def register(self, mcp: Any) -> None:
    @mcp.tool()
    def my_component_status() -> dict:
        return {"success": True, "status": "ok"}
```

返回值建议统一为字典，成功时包含 `success: True`，失败时包含 `success: False` 和 `error`。现有组件中 `exa`、`local_schedule`、`windows_manager` 都采用这种模式。

## `invoke_tool()` 规约

如果组件需要在 client 侧或集群远程调用中运行，实现 `invoke_tool()`：

```python
def invoke_tool(self, tool_name: str, arguments: dict[str, Any]) -> dict:
    if tool_name == "my_component_status":
        return {"success": True, "status": "ok"}
    raise RuntimeError(f"unknown tool: {tool_name}")
```

建议 `register()` 和 `invoke_tool()` 复用同一业务逻辑，避免服务端和客户端行为分叉。

## 配置读取

组件配置段名称必须与组件名一致：

```toml
[my_component]
enabled = true
timeout_seconds = 20
```

工厂函数可以这样读取：

```python
def build_component(config: dict[str, Any] | None = None, full_config: dict[str, Any] | None = None) -> MyComponent:
    section = config or {}
    return MyComponent(
        timeout_seconds=int(section.get("timeout_seconds", 20)),
        full_config=full_config,
    )
```

敏感信息优先支持环境变量。例如 `exa` 支持 `EXA_API_KEY`，同时也允许在 `[exa]` 中配置 `api_key`。

## 更新 `index.json`

新增组件必须追加索引项：

```json
{
  "name": "my_component",
  "description": "Short one-line description.",
  "version": "0.1.0",
  "path": "my_component",
  "entry": "component.py",
  "readme": "README.md",
  "platforms": ["Windows", "Linux", "MacOs"]
}
```

字段要求：

| 字段 | 要求 |
| --- | --- |
| `name` | 与目录名一致，只使用字母、数字、下划线或短横线 |
| `description` | 一句话说明组件用途 |
| `version` | 语义化版本，例如 `0.1.0` |
| `path` | 相对仓库根目录的组件路径 |
| `entry` | 入口文件，默认 `component.py` |
| `readme` | 文档文件，默认 `README.md` |
| `platforms` | 支持平台数组，值为 `Windows`、`Linux`、`MacOs` |

## README 结构

每个组件 README 建议使用固定结构：

```markdown
# my_component

一句话说明组件做什么、适合什么场景。

## Tools

- `my_component_action(param)` — 工具说明。

## Config

```toml
[my_component]
enabled = true
```

## Dependencies

- 外部 Python 包或系统依赖

## Usage Notes

- 行为限制、平台限制和常见问题

Platforms: Windows|Linux|MacOs
```

最后一行 `Platforms: ...` 是目录系统解析平台兼容性的依据，不要在它后面追加任何内容。

## 错误处理建议

平台或依赖不可用时，避免在 import 阶段直接崩溃。`windows_manager` 的做法是延迟导入 `pywin32`，在工具调用时返回清晰错误：

```python
RUNTIME_ERROR_MESSAGE = "windows_manager is only available on Windows with pywin32 installed (pip install pywin32)."

def _load_win32_modules() -> tuple[Any, Any, Any]:
    try:
        import win32con
        import win32gui
        import win32process
    except Exception as exc:
        raise RuntimeError(RUNTIME_ERROR_MESSAGE) from exc
    return win32con, win32gui, win32process
```

这种模式可以保证非目标平台仍能安全扫描组件目录。

## 测试建议

组件逻辑应优先拆出无外部依赖的纯函数或数据层，再编写测试。`local_schedule` 的测试使用临时 SQLite 文件：

```python
def test_add_deadline_event_uses_due_time_for_range_fields(tmp_path):
    store = LocalScheduleStore(db_path=str(tmp_path / "schedule.sqlite3"))
    event = store.add_event(
        title="Ship release",
        schedule_type="deadline",
        due_time="2026-05-10T12:00:00Z",
    )
    assert event["schedule_type"] == "deadline"
    assert event["due_time"] == "2026-05-10T12:00:00"
```

运行测试：

```bash
pytest
```

运行静态检查：

```bash
ruff check .
```

## 发布检查清单

- 组件目录包含 `__init__.py`、`component.py`、`README.md`。
- `component.py` 实现 `register()` 和 `export_tools()`。
- 需要远程调用时实现 `invoke_tool()` 或 `ainvoke_tool()`。
- 工具名带组件前缀且全局唯一。
- `index.json` 已追加组件条目，平台字段与 README 最后一行一致。
- README 包含工具、配置、依赖、使用说明和平台行。
- 平台依赖延迟加载，非目标平台可安全 import。
- 有可运行的测试或明确说明无法自动化测试的系统依赖。
